# Harness Engineering（规约框架 · 多适配容器）

跨技术栈 AI 编程流程规约。本仓库是**多适配容器**：同一套 Harness Engineering 通用规约，按不同编辑器/Agent 工具各维护一份**以自身目录为根、彼此平级、互不耦合**的完整适配。

## 适配一览

| 适配目录 | 说明 | 状态 |
| -------- | ---- | ---- |
| `cursor/` | 适配 Cursor 的完整规约（自动 Hook 门禁、7 个子角色、脚本、模板、E2E / 接口测试 R14 / 存储对账 R17 / QE 硬门禁 R15/R16、架构/代码通用原则） | 可用 |
| `trae/` | 适配 Trae 的完整规约（原生 Hook + 手动 `gate-check` 兜底、7 个 Subagent、脚本、模板、E2E / 接口测试 R14 / 存储对账 R17 / QE 硬门禁 R15/R16、架构/代码通用原则） | 可用 |

> **各适配共享要点**（细节见各适配 `AGENTS.md` 与自带 README，此处仅作容器层速览）：`detail-design-spec.md` 定义架构设计原则（§2）与代码编写通用原则（§5）、安全编码要求（§8）；**需求评审专家**在设计阶段审核需求覆盖度、目标达成性与架构原则，系统架构师产出时须遵循，且设计问题清单经 **R18** 机读（12 维 + 可修复字段 + P0 需求覆盖矩阵）后方可进入开发；QE 阶段含 R15 编程规范 lint 与 R16 静态代码质量（重复代码 + 密钥泄露扫描）硬门禁；批次集成测试含 **R17 业务数据存储对账**——凡涉及业务数据持久化（数据库 / 文件 / 缓存 / 对象存储等），接口测试与 Playwright E2E 均须在报告「## 存储对账记录」留痕并机读过门禁，禁止仅以 HTTP 200 或界面无报错判定通过；无持久化项目走双要素豁免（权威定义见各适配 `AGENTS.md` §8.3）。

## 前置条件

- 各适配的门禁与初始化脚本依赖 **`Node.js >= 18`**（执行 `.mjs`；R16 静态扫描亦经 `npx` 跨技术栈获取工具）。
- 目标项目的业务技术栈不限；具体运行时、包管理器与测试工具由系统架构师在设计阶段声明。

## 工作区根（重要）

仓库根 `easy-harness/` 只是容器（`.git` 在此），**本身不作为工作区根**。使用某一适配时，二选一：

- 直接把对应适配目录（如 `easy-harness/cursor/`、`easy-harness/trae/`）作为工作区根打开；或
- 把该适配目录的**全部内容**复制到宿主项目根，再以宿主项目为工作区根。

> **切勿**把外层 `easy-harness/` 当作工作区根——容器层没有具体适配的机件（如 `.cursor/`、`.trae/`、`AGENTS.md`），工具会找不到门禁与子角色定义，整套规约将**静默失效**。

## 未来扩展

后续需要适配新的编辑器/Agent 工具时，无需改动已有适配，按如下方式新增即可：

1. 在仓库根**新增一个与 `cursor/` 平级的目录**（目录名取该工具名，如 `<tool>/`）。
2. 在该目录下实现一套**以自身为根、完整自洽**的适配——对照 `cursor/` 的结构，落地该工具自身的规约载体与机件（顶层规约文件、门禁/Hook 机制、子角色定义、脚本、模板，以及描述其运行时产物的 `.gitignore`）。具体拆分哪些角色、用何种门禁机制，由该工具的能力决定，不必与 `cursor/` 一一对应。
3. 该适配同样以「作为工作区根打开」或「整体复制到宿主项目根」两种方式使用；其 `.gitignore` 随适配目录一并复制，确保运行时产物在宿主项目也不被提交。

> 各适配自成一体、互不引用（含各自的 `.gitignore`），因此新增或调整某个适配不会影响其它适配。
